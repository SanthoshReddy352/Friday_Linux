# FRIDAY Kali Workflow Planner â€” Implementation Plan

## Context

The user is adding a Kali Linux cybersecurity workflow capability to FRIDAY, controllable from both desktop and Telegram (already wired as the mobile channel). Three research reports (`research/Kali_Research_Perplexity.md`, `Kali_Research_Deepseek.md`, `Kali_Implementation_Gemini.md`) converged on the same architecture:

- **Qwen3.5-4B (the existing tool model)** stays a *bounded* planner â€” intent classification, workflow selection, slot fill, plan draft, observation summary. **Never raw shell.**
- All execution flows through **safe capability wrappers** that own command templates and shlex-escape args.
- **YAML workflow templates** for known multi-step tasks (lab inventory, web recon, DNS enum, etc.).
- **Plan validator** with strict JSON schemas (IntentClassification, ToolPlan, WorkflowPlan, Observation, ReplanDecision).
- **Permission/consent gates** for write/critical/networked actions, with approval dialog over Telegram.
- **Bounded observationâ†’replanning loop** (max steps, retries, timeouts).
- **Procedural memory** of approved successful plans, retrieved as few-shot exemplars.

### Decisions captured

| Decision | Choice |
|---|---|
| Scope | Full roadmap, Phases 1â€“8 |
| Sandboxing (v1) | `shlex.quote` + flag deny-list + `PermissionService` scope gates. Upgrade to firejail/Docker later if needed. |
| JSON decoding (v1) | Soft JSON mode (current llama-cpp `{type: json_object}`) + Pydantic validation + `json_repair` + 1 retry. Add grammar constraints (Outlines/XGrammar) later only if real-world validation failure rate >5%. |
| Phase 1 toolset | nmap (host_service_scan + ping_sweep), gobuster/ffuf (web_directory_enum), dig (dns_enum_owned_domain), deterministic report_generator + scan_diff |
| Mobile channel | Existing Telegram pipeline (bidirectional, `source="telegram"` already first-class). Extend with approval dialog + progress streaming. |
| Model | Existing Qwen3.5-4B tool model (config.yaml:120). No model swap. |

### Critical gaps to fill

1. No Kali tool wrappers.
2. No YAML workflow template loader/registry.
3. Capability descriptor missing security fields (`allowed_use_cases`, `forbidden_use_cases`, `network_scope`, `requires_authorization`, `command_templates`, `argument_constraints`, `parser`, `next_step_hints`, `rollback_or_cleanup`).
4. No structured plan-draft / observation / replan JSON schemas as first-class Pydantic models.
5. No Telegram approval round-trip (ConsentService is currently always-allow).
6. No mid-task progress streaming to Telegram.
7. No procedural plan archive / few-shot retrieval for the LLM planner.
8. No deterministic output parsers for Kali tools (nmap XML, gobuster JSON, dig text).

## Phased implementation

Each phase is shippable on its own. The first concrete vertical slice (Phase 1 + a single nmap wrapper + a `lab_network_inventory` workflow) proves the pattern end-to-end before broadening.

### Phase 1 â€” Capability descriptor hardening + first Kali wrapper

**Goal:** Extend the `CapabilityDescriptor` schema with the security-required fields, register a single nmap wrapper end-to-end (intent â†’ broker â†’ executor â†’ parser â†’ response), prove shlex+permission gating works.

**Files to modify:**
- `core/capability_registry.py` â€” add fields to `CapabilityDescriptor`: `allowed_use_cases`, `forbidden_use_cases`, `network_scope` (local/lab/public/unknown), `requires_authorization`, `command_templates` (dict[modeâ†’template]), `argument_constraints` (dict), `parser` (str name), `success_conditions`, `failure_conditions`, `next_step_hints`, `rollback_or_cleanup`, `logging_requirements`. All optional with safe defaults so existing 20+ capabilities keep working.
- `core/kernel/permissions.py` â€” add `check_network_scope(target, allowed_scope)` and `check_authorized_target(target_value, scope_id)`.
- `config.yaml` â€” add `security:` section with `authorized_scopes` (list of CIDR/domains) and `lab_mode: true`.

**Files to create:**
- `modules/security_tools/__init__.py`
- `modules/security_tools/plugin.py` â€” registers `host_service_scan`, `ping_sweep` capabilities.
- `modules/security_tools/wrappers/nmap_wrapper.py` â€” class `NmapWrapper` with `host_service_scan(target, profile, ports)` and `ping_sweep(subnet)`. Builds command via `shlex.quote`, runs in subprocess with timeout, returns raw bytes.
- `modules/security_tools/parsers/nmap_parser.py` â€” `parse_nmap_xml(xml_bytes) -> Observation`. Use `python-libnmap` if available, else stdlib `xml.etree`.
- `modules/security_tools/safety.py` â€” `validate_target_scope(target, run_context)`, `block_dangerous_flags(args)`.
- `tests/test_security_tools_nmap.py` â€” unit tests against canned nmap XML fixtures; integration test with `127.0.0.1`.

**Verification:**

- `pytest tests/test_security_tools_nmap.py -v`

### Phase 3 â€” Qwen JSON planner prompts

**Goal:** Replace the current single planner prompt with seven structured prompts (intent classification, workflow selection, slot-fill, plan draft, plan validation, observation summary, replan decision) â€” each tied to a Pydantic schema.

**Files to create:**
- `core/planning/schemas.py` â€” Pydantic models: `IntentClassification`, `ToolPlanV2` (extends current ToolPlan), `WorkflowPlan`, `Observation`, `ReplanDecision`.
- `core/planning/prompts/` â€” Markdown/Jinja templates: `intent_classification.j2`, `workflow_selection.j2`, `slot_fill.j2`, `plan_draft.j2`, `plan_validate.j2`, `observation_summary.j2`, `replan.j2`. Each includes safety policy, compact capability cards, few-shot examples.
- `core/planning/qwen_planner.py` â€” `QwenPlanner` class wrapping `LocalModelManager`. One method per prompt. Each method: render prompt â†’ call model with `json_response=True` â†’ `json_repair` on failure â†’ Pydantic validate â†’ 1 retry with error feedback â†’ raise on second failure.

**Files to modify:**
- `core/capability_broker.py` â€” when deterministic confidence is low, route to `QwenPlanner.draft_plan()` instead of the current single ModelRouter call.
- `modules/security_tools/plugin.py` â€” register *compact* capability cards (selector hint, required slots, risk, network scope) for prompt injection. Full schemas only injected once the capability is selected.

**Verification:**
- `pytest tests/test_qwen_planner.py -v` (mocked model output, validates schema enforcement, JSON repair, retry behavior).
- Ambiguous turn: "find open services on my CTF box" â†’ log shows `clarify` intent with `missing_slots=["target_host", "ctf_scope_confirmation"]`.

### Phase 4 â€” Plan validator + deterministic repair

**Goal:** Every plan (template-compiled or LLM-drafted) passes through a deterministic validator before execution. Validator is authoritative for safety; the LLM is never trusted for permission decisions.

**Files to create:**
- `core/planning/plan_validator.py` â€” `PlanValidator.validate(plan, run_context, registry) -> ValidationResult`. Checks: all `capability` names exist in registry, `args` match `input_schema`, `depends_on` references are valid and acyclic, `network_scope` is consistent with `run_context.authorized_scope`, no step exceeds the user's `risk_level` ceiling, deny-listed flags absent from `argument_constraints`.
- `core/planning/plan_repair.py` â€” `PlanRepair.try_repair(plan, errors)` for fixable issues (missing nulls, enum normalization, extraneous keys). Unfixable issues bubble up.

**Files to modify:**
- `core/planning/turn_orchestrator.py` â€” insert validator between `PlannerEngine.plan()` and executor dispatch.

**Verification:**
- Synthetic plan with cycle â†’ validator rejects with clear error.
- Plan referencing public IP when scope is `lab` â†’ blocked.
- `pytest tests/test_plan_validator.py -v`.

### Phase 5 â€” Observation parsing + structured tool output

**Goal:** Every Kali tool wrapper has a deterministic parser that converts raw stdout/stderr into a typed `Observation` object. The LLM only ever sees the structured observation, never raw terminal noise.

**Files to create:**
- `modules/security_tools/parsers/` â€” `nmap_parser.py` (XML), `gobuster_parser.py` (JSON output mode), `dig_parser.py` (text), `report_parser.py` (markdown), `scan_diff_parser.py`.
- `core/planning/observation.py` â€” canonical `Observation` model (already in Phase 3 schemas).

**Files to modify:**
- `core/tool_execution.py` / `core/task_graph_executor.py` â€” after subprocess returns, if the capability descriptor has `parser` set, route the raw output through it and store the structured `Observation` in run context for downstream steps and the replanner.

**Verification:**
- Run nmap against `127.0.0.1`, check `run_context.observations["s1"].structured_data.open_ports` is a list of ints.
- `pytest tests/test_security_parsers.py -v`.

### Phase 6 â€” Bounded observation â†’ replanning loop

**Goal:** When a step returns `failure`/`partial`/`timeout`, the orchestrator decides via deterministic rules first; only escalates to `QwenPlanner.replan()` for genuine ambiguity. Hard caps on retries and total steps.

**Files to create:**
- `core/planning/replan_controller.py` â€” `ReplanController.decide_next(state, observation) -> ReplanDecision`. Deterministic rules: timeout once â†’ retry; out-of-scope â†’ refuse; missing dependency â†’ ask user; unknown â†’ escalate to Qwen.

**Files to modify:**
- `core/workflow_orchestrator.py` â€” wire `ReplanController` into `TemplateWorkflow` and the v2 turn loop.
- `config.yaml` â€” `routing.max_workflow_steps`, `routing.max_step_retries`, `routing.workflow_total_timeout_sec`.

**Verification:**
- Force an nmap timeout (firewalled host) â†’ ReplanDecision is `retry` with bumped timeout; on second timeout, `stop` with partial summary.
- `pytest tests/test_replan_controller.py -v`.

### Phase 7 â€” Telegram approval + progress streaming

**Goal:** Workflows can ask the user yes/no over Telegram before risky steps, and stream "step k/N complete" updates during long runs. Restores ConsentService for `requires_authorization: true` capabilities.

**Files to modify:**
- `core/kernel/consent.py` â€” un-stub `ConsentService.evaluate()` for the new `network_scope in {lab,public}` and `side_effect_level in {write,critical}` cases. Return `pending_approval` when channel-bound approval is needed.
- `modules/comms/telegram.py` â€” add `request_approval(question, options=["approve","deny","cancel"], timeout=180) -> str`. Implementation: send message, set a per-chat pending future, the next inbound message matching the options resolves it.
- `modules/comms/plugin.py` â€” new tool `send_progress(text)` that uses `current_turn().source` to pick channel.
- `core/capability_broker.py` â€” when consent returns `pending_approval` and `current_turn().source == "telegram"`, hand off to `comms.telegram.request_approval`. For desktop/voice, keep the existing voice consent flow.
- `core/task_graph_executor.py` â€” emit a `step_completed` event after each wave; new `TelegramProgressSubscriber` in `modules/comms` listens and streams.

**Verification:**
- Telegram turn: "scan 192.168.56.0/24 for services" â†’ bot replies with plan preview + "Approve?" buttons â†’ user taps Approve â†’ progress messages stream â†’ final report posts.
- `grep -i "request_approval\|send_progress" logs/friday.log | tail -20`
- Manual test: send Telegram message, deny approval, verify workflow cancels and no shell command was issued (audit log empty for that turn).

### Phase 8 â€” Procedural plan archive + few-shot retrieval

**Goal:** Store user-approved successful plans; retrieve top-k similar exemplars and inject them into the Qwen planner prompts to improve consistency.

**Files to create:**
- `core/memory/plan_archive.py` â€” `PlanArchive.save(plan, outcome, user_id)` and `.retrieve_similar(user_text, top_k=3)`. Embed user requests with the existing sentence-transformer used by the semantic memory; filter by `outcome == "success"` and `user_approved == True`.

**Files to modify:**
- `core/memory_broker.py` â€” `build_bundle()` includes top-3 similar prior plans when intent type is `workflow` or `multi_step`.
- `core/planning/prompts/*.j2` â€” slot in `{{retrieved_examples}}` where the schemas already expect it.
- `core/workflow_orchestrator.py` â€” on workflow completion, call `PlanArchive.save()`.

**Verification:**
- Run `lab_network_inventory` twice. On the second turn, log shows the first run was retrieved and injected as a few-shot.
- `sqlite3 data/friday.db "SELECT workflow_name, outcome, user_approved FROM plan_archive ORDER BY id DESC LIMIT 5;"`
